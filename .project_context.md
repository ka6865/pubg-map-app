# BGMS 프로젝트 컨텍스트 및 작업 가이드
*최종 업데이트: 2026-06-09 (3D 리플레이 바닥 격자 차폐 정합 및 지형 불투명화 적용)*

이 문서는 BGMS 프로젝트에서 AI와 개발자가 빠르게 전체 흐름을 이해하고, 변경 지점을 정확히 찾기 위한 기준 문서입니다. 실제 코드를 우선 기준으로 삼고, 이 문서는 작업 방향과 주요 연결 관계를 보조합니다.

## 1. 프로젝트 정체성

BGMS는 PUBG 유저를 위한 전술 지도, 전적 검색, 텔레메트리 리플레이, 전술 분석, AI 코칭을 제공하는 Next.js 애플리케이션입니다.

핵심 사용자 흐름은 다음과 같습니다.

```text
루트 접속
-> /maps/erangel 기본 리다이렉트
-> 지도/마커/핫드랍/제보/시뮬레이터 사용
-> 전적 검색 시 PUBG API 조회
-> 매치별 AnalysisEngine 분석
-> Cloudflare R2 하이브리드 캐시 및 Supabase DB 저장
-> 통계 UI, 지도 리플레이, AI 요약/코칭에서 재사용
```

## 2. 현재 기술 스택

- 프레임워크: Next.js 16.1.1, App Router
- UI: React 19.2.3, React DOM 19.2.3
- 지도: Leaflet, React Leaflet, leaflet.heat
- 스타일: Tailwind CSS v4, 전역 CSS 변수 기반 테마
- 백엔드/인프라: Supabase Database, Auth, Realtime, Cloudflare R2 (AWS S3 SDK)
- AI: AI SDK, Google Generative AI
- 모바일: Capacitor v8
- 테스트: Jest, Vitest

AI 모델 fallback 풀은 `app/api/pubg/ai-analyze/route.ts`와 `app/api/pubg/ai-summary/route.ts`의 실제 구현을 우선 확인합니다. 현재 중요한 fallback 예시는 아래와 같습니다.

```ts
const modelsToTry = [
  "gemini-3.1-flash-lite",
  "gemini-3-flash-preview",
  "gemini-2.5-flash",
];
```

모델명 AI 에이전트는 최신화가 되어있지않으니 사용자 외에 변경은 비용, 응답 형식, 스트리밍 안정성 영향 및 없는 모델일수도있으니 절대 변경하지 않습니다.
추가를 하더라도 위에 풀백 로직은 유지하며 모델명은 개발자가 최신화 할수있도록합니다.

## 4. 핵심 데이터 파이프라인

### 플레이어 검색

`app/api/pubg/player/route.ts`

1. PUBG API에서 닉네임으로 accountId를 조회합니다.
2. 사용 가능한 시즌 목록과 최근 매치 ID를 가져옵니다.
3. 랭크/일반 시즌 통계를 함께 조회합니다.
4. FPP/TPP는 `roundsPlayed`가 많은 쪽을 대표 모드로 선택합니다.
5. 네트워크 오류는 `withRetry`로 재시도합니다.

### 매치 분석

`app/api/pubg/match/route.ts`

1. `matchId`, `nickname`, `platform`을 받아 PUBG Match API를 조회합니다.
2. 참가자, 로스터, 내 팀 정보를 계산합니다.
3. **[ISR V1.0]** `unstable_cache` 프록시를 통해 `processed_match_telemetry`에서 분석 결과 캐시를 조회합니다. Cache Hit 시 DB 커넥션 0건으로 즉시 반환됩니다.
4. 캐시가 없으면 텔레메트리를 가져옵니다. (RESULT_VERSION 버전 비교 로직은 ISR 도입으로 제거되었습니다.)
5. 텔레메트리는 분석에 필요한 이벤트와 필드만 남겨 **Cloudflare R2 스토리지(`bgms` 버킷)**에 업로드하여 캐싱합니다 (`r2Service.ts` 활용).
6. `AnalysisEngine`을 실행해 전술 지표를 계산합니다.
7. 대용량 지도 리플레이 데이터(`mapData`)는 **R2 스토리지**에 업로드하고, DB 테이블 `match_master_telemetry`의 `storage_path` 컬럼에 R2의 실물 파일명(`map.json`)을 매핑(도킹)합니다. 요약/전술 결과는 DB에 저장합니다.
8. `/api/pubg/ingest`를 백그라운드로 호출해 벤치마크 데이터 축적을 트리거합니다.

캐시 관리는 두 가지 레이어로 구분합니다.

- **ISR 캐시 (Next.js 16 unstable_cache + revalidateTag)**: `match/route.ts`의 DB 읽기를 `unstable_cache`로 래핑하여 Vercel Edge 캐시에 7일간 보존합니다. 배포 시 `app/api/admin/revalidate` 엔드포인트를 호출하여 `revalidateTag('match-analysis', 'max')`로 즉시 만료시킵니다.
- **R2 파일 캐시**: `TELEMETRY_VERSION`이 R2 파일명에 포함되어 파일 수준의 캐시 무효화를 수행합니다. (ISR과 무관)
- `RESULT_VERSION`: DB 레코드에 기록되는 엔진 버전 식별자입니다. ISR 도입 이후 수동 범핑은 대규모 엔진 변경 시에만 수행합니다.

### 텔레메트리 리플레이

`app/api/pubg/telemetry/route.ts`와 `hooks/useTelemetry.ts`

1. 지도에서 `?playback={matchId}&nickname={nickname}` 형태로 리플레이를 요청합니다.
2. `useTelemetry`가 `/api/pubg/telemetry`를 호출합니다.
3. **R2 스토리지**에 저장된 `{matchId}_{nickname}_v{TELEMETRY_VERSION}_map.json` 캐시를 `downloadFromR2`를 통해 우선 다운로드하여 사용합니다.
4. 캐시가 없으면 PUBG 텔레메트리를 직접 받아 `AnalysisEngine`을 실행하고 `mapData`를 생성하여 R2에 업로드합니다.
5. `TelemetryCanvasLayer`, `PlayerPathRenderer`, `PlayerMarkerRenderer`, `CombatRenderer`, `ShotRenderer`, `ZoneRenderer`가 지도 위에 재생 데이터를 렌더링합니다.
6. 3D 리플레이(`app/replay/3d/page.tsx`) 상에서도 2D 지도와 일관되게 맵별 하이트맵 포맷에 맞춰 고밀도 고도 모델을 복구(하이브리드화)하였습니다. 기존 PNG 파일(에란겔, 미라마, 비켄디)은 브라우저 Canvas 디코더의 sRGB 감마 2.2 보정을 역산하기 위해 `const R_linear = Math.pow(R / 255, 2.2) * 255;` 역 감마 보정을 수행하고, 맵별 16비트 Unreal Engine 물리 스케일과 정합되는 수식 `(R_linear - zOffset) * zScale`을 적용하였습니다. 특히 미라마의 경우 높은 고지대의 오차를 배제하기 위해 미라마 전용 오프셋과 스케일(`zScale = 3.0`, `zOffset = 135`)을 적용하여 저지대(페카도)와 고지대(산 정상) 모두에서 텔레메트리 절대 고도와 1.5m 이내로 일치시켰습니다. 다른 PNG 맵은 표준 오프셋 128과 표준 Z스케일 `2.048`을 공통 적용합니다. JPG 파일(론도, 태이고, 데스턴)은 `MAP_ELEVATION_CONFIGS` 기반의 정규화 공식을 사용하여 정합시켰습니다. 특히 3D 지형 메쉬의 X축 -90도 회전(Local Y -> World -Z)에 따른 세로축(남북 방향) 기하학적 매핑에 대해, 비동기 로딩되는 그리드 캐시 조회 시 Z축 반전 계산 오류(`1 - percentZ`)를 제거하여 폴백 방식 픽셀 쿼리와의 1:1 기하학적 정합성을 바로잡았습니다. 또한 지형 메쉬의 기하학적 보간 오차를 방지하기 위해 **이중 선형 보간(Bilinear Interpolation) 격자 고도 캐시(`terrainGridRef`)**를 사용합니다. 나아가 지상 스냅 알고리즘을 지능적으로 고도화하여 캐릭터가 땅속으로 묻히는 지하 매몰 현상을 즉각 차단(diffM < 0일 때 무조건 스냅)하고, 일반 차량 탑승 상태(isInVehicle) 시에는 급경사 오차를 커버하기 위해 25m 완화 스냅을 적용하며, 도보 시에는 옥상 교전 분리 가독성을 위해 8m 완화 스냅을 동적으로 교차 적용하는 다단계 지상 안착 필터링을 구축하였습니다. 이를 통해 산악 지형의 급사면에서 캐릭터와 버기 등의 탑승 장비가 하늘로 붕 뜨는 비주얼 버그를 완전히 종식시키고, 건물 옥상의 입체 고도 시각화를 함께 보존하였습니다. 또한 미지원 소형 4개 맵(Sanhok, Karakin, Paramo, Haven)을 정돈하여 대형 6개 맵(Erangel, Miramar, Vikendi, Taego, Deston, Rondo)으로 지원 범위를 통일하였습니다. 더불어, 지형 메쉬 아래 바닥에 깔린 격자선(그리드, GridHelper)이 특정 저지대(음수 고도 지역) 지면 위로 뚫고 나와 붕 뜨는 현상을 해결하기 위해, 지형 메쉬 재질의 투명도를 제거(transparent: false)하고, 하이트맵 로딩 콜백 내에서 지형의 최저 고도(minElevation) 값을 동적으로 계산하여 격자망의 Y 좌표를 최저 고도보다 살짝 아래로 자동 동화(minElevation - 0.05)시켰습니다. 이를 통해 맵의 어떤 저지대 골짜기에서도 격자가 지상으로 튀어나오는 물리적 간섭 결함을 완전히 종식시켰습니다.
7. 2D 지도뷰(`components/map/MapView.tsx`) 내 박격포 및 핫드랍 등에 활용되는 고도 계산 역시 동일한 역 감마 보정 및 미라마 전용 오프셋/스케일을 적용하여 3D 뷰어와 완전히 동일하게 정합시켰습니다.

 리플레이 성능은 이벤트 수, Canvas 렌더링, 상태 보간 로직에 크게 좌우됩니다. 대량 데이터 변경 시 `useTelemetry`와 `components/map/telemetry/*`를 함께 확인합니다. **3D 리플레이 지도 텍스처**는 기존 zoom 0 단일 256×256 이미지 대신, `buildHighResTileTexture` 함수가 zoom 2의 4×4 타일 16장을 비동기로 병렬 로드하여 Canvas에 합성(1024×1024)한 후 Three.js Texture로 변환하여 적용합니다. GPU의 최대 이방성 필터링(anisotropy)도 함께 적용해 경사 뷰에서도 선명한 화질을 유지하며, 합성 실패 시 zoom 0으로 폴백합니다.

### AI 코칭/요약

- `app/api/pubg/ai-analyze/route.ts`: 단일 매치 기반 코칭 리포트입니다.
- `app/api/pubg/ai-summary/route.ts`: 여러 매치 집계 기반 최근 플레이 요약입니다.

두 API 모두 분석 결과를 사람이 읽기 좋은 프롬프트 데이터로 변환한 뒤 Gemini 계열 모델에 전달합니다. 응답은 JSON 또는 NDJSON 스트리밍 형태로 소비될 수 있으므로, 프롬프트 구조와 클라이언트 파싱 로직을 함께 검토해야 합니다.

## 5. 주요 폴더 역할

### `app/`

Next.js App Router의 라우트와 API가 모여 있습니다.

- `layout.tsx`: 전역 레이아웃, 메타데이터, AuthProvider 래핑
- `globals.css`: Tailwind v4 및 전역 테마 변수
- `actions/map-settings.ts`: 맵 설정 관련 Server Action
- `api/pubg/*`: PUBG API, R2 텔레메트리 연동, AI 분석, 벤치마크 관련 백엔드
- `api/admin/*`: 관리자 승인/거절/게임 데이터 관리
- `api/cron/*`: 핫드랍, 패치노트 예약 작업

### `components/`

UI와 클라이언트 상호작용 컴포넌트가 모여 있습니다.

- `components/map/*`: 전술 지도, 도구, 제보, 시뮬레이터, 텔레메트리 재생
- `components/map/telemetry/*`: 리플레이 전용 Canvas/레이어 렌더러
- `components/stat/*`: 전적 분석 카드, 차트, AI 요약, 매치 타임라인
- `components/board/*`: 게시판 UI
- `components/admin/*`: 관리자 편집 UI
- `components/common/*`: 글로벌 헤더, 모바일 메뉴, 하단 내비게이션, 푸터

### `hooks/`

클라이언트 비즈니스 로직을 캡슐화합니다.

- `useTelemetry.ts`: 리플레이 데이터 요청, 재생 시간, 플레이어 상태 보간
- `useMapData.ts`: 맵별 마커/제보 데이터 관리
- `useMapSettings.ts`: 맵 카테고리/아이콘 설정 동기화
- `useRealtimeToast.ts`: Supabase Realtime 기반 알림

### `lib/`

공유 설정, 분석 엔진, 도메인 유틸리티가 모여 있습니다.

- `lib/pubg-analysis/AnalysisEngine.ts`: 전술 분석 엔진 진입점
- `lib/pubg-analysis/handlers/*`: 전투, 위치, 자기장, 유틸리티, 리플레이 처리
- `lib/pubg-analysis/r2Service.ts`: Cloudflare R2 버킷 업로드/다운로드/배치 삭제 인터페이스
- `lib/pubg-analysis/constants.ts`: 분석/텔레메트리 버전, 맵/무기 상수, 임계값
- `lib/pubg-analysis/types.ts`: 분석 결과와 내부 상태 타입
- `lib/map_config.ts`: 맵 좌표, 타일, 아이콘, 좌표 변환 설정
- `lib/seo-config.ts`: 동적 SEO 설정
- `lib/supabase.ts`, `lib/supabase-middleware.ts`: Supabase 클라이언트/세션 보조

### `utils/supabase/`

Next.js 서버/클라이언트 환경에 맞춘 Supabase SSR 클라이언트 생성 유틸리티입니다.

## 6. 변경 작업 시 우선 확인할 것

### 분석 엔진 수정

`lib/pubg-analysis/*`를 수정할 때는 아래 호출부를 같이 확인합니다.

- `app/api/pubg/match/route.ts`
- `app/api/pubg/telemetry/route.ts`
- `app/api/pubg/ingest/route.ts`
- `app/api/pubg/ai-summary/route.ts`
- `app/api/pubg/ai-analyze/route.ts`
- `components/stat/*`
- `hooks/useTelemetry.ts`

분석 결과 타입, `mapData` 구조, 캐시 버전이 바뀌면 `RESULT_VERSION` 또는 `TELEMETRY_VERSION` 증가가 필요할 수 있습니다.

### R2 스토리지 및 클린업 규칙

- 대용량 텔레메트리 가공 원본 데이터는 Supabase DB에 적재하지 않고, Cloudflare R2 버킷 `bgms`에 저장합니다.
- `scripts/cleanup_telemetry.ts` 스크립트를 사용해 고립(Orphaned) 찌꺼기 파일을 지울 때는, R2 API의 벌크 소각(`deleteMultipleFromR2`) 명령을 사용하여 한 번에 최대 1000개씩 초고속으로 정리합니다.
- R2 서비스 임포트 시 Next.js 절대 경로(`@/`)를 사용할 수 없는 단독 독립 실행 스크립트(scripts/* 등)에서는 반드시 상대 경로(`../lib/pubg-analysis/r2Service`)를 유지해야 오류가 발생하지 않습니다.

### 텔레메트리 필드

PUBG 텔레메트리 필드는 `.pubg-telemetry-guide.md`를 우선 참고합니다.

- `maker`, `dBNOMaker`: 기절/DBNO 원인 추적
- `killer`, `finisher`: 최종 사살/마무리 추적
- `reviver`: 부활/소생 추적
- `attacker`, `victim`: 피해/교전 이벤트 추적
- `dBNOId`: 기절과 확킬/트레이드 인과관계 연결

필드명을 단순 추측으로 바꾸지 않습니다. `LogPlayerKillV2`, `LogPlayerMakeGroggy`, `LogPlayerTakeDamage`, `LogPlayerRevive`는 서로 의미가 다릅니다. (※ 구버전인 `LogPlayerKill` 대신 실제 PC 텔레메트리에서는 `LogPlayerKillV2`가 100% 독점적으로 인입됩니다.)

### 지도/좌표 수정

지도 좌표나 타일 처리는 `lib/map_config.ts`, `components/map/MapView.tsx`, `components/map/telemetry/*`를 함께 봅니다.

PUBG 원본 좌표, 이미지 픽셀 좌표, Leaflet 좌표가 섞여 있으므로 다음을 반드시 확인합니다.

- 맵 크기: 8192, 4096, 3072, 2048 계열
- 원본 텔레메트리 좌표의 cm 단위 변환
- 맵별 y축 보정
- 리플레이 좌표와 일반 마커 좌표의 기준 일치

### Supabase/보안 수정

관리자 기능은 클라이언트 화면의 가드만으로 끝내지 않습니다. `app/api/admin/*` 라우트에서 반드시 서버 측 세션과 `profiles.role === "admin"` 검증을 수행합니다.

Service Role Key는 서버 라우트에서만 사용합니다. 클라이언트 컴포넌트에 노출하지 않습니다.

### RSC/Client 분리

페이지는 가능한 Server Component로 유지하고, 브라우저 API, Leaflet, localStorage, 상태/이벤트가 필요한 부분만 Client Component로 분리합니다. 지도와 통계 UI는 클라이언트 로직이 많으므로 import 경계를 특히 조심합니다.

## 7. 코드 품질 규칙

- **함수/인터페이스 호출부-선언부 동기화**: 함수나 타입을 신설/수정할 때, 호출하는 곳(Caller)과 선언부(Callee)의 타입이 완벽히 일치하는지 전수 검사합니다 (Zero-Error Policy).
- **Dead Code 및 임시 더미 소각**: 사용하지 않는 import, 주석 처리된 코드, 의미 없는 `console.log` 및 하드코딩된 더미 데이터는 발견 즉시 즉각 삭제하거나 실시간 Supabase 연동(`actions/`) 로직으로 교체합니다.
- 대용량 텔레메트리 데이터는 DB 행에 직접 넣기보다 Cloudflare R2 버킷 캐시를 우선하여 사용합니다.
- 프롬프트/AI 응답 JSON 구조를 바꾸면 클라이언트 파싱과 에러 fallback을 함께 수정합니다.
- 성능에 민감한 지도/리플레이 변경은 데스크톱과 모바일 레이아웃을 모두 확인합니다.

rg "r2Service|uploadToR2|downloadFromR2"

rg "AnalysisEngine"
rg "RESULT_VERSION|TELEMETRY_VERSION"
rg "playback"
