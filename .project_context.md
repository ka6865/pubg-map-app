# BGMS 프로젝트 컨텍스트 및 작업 가이드

이 문서는 BGMS 프로젝트에서 AI와 개발자가 빠르게 전체 흐름을 이해하고, 변경 지점을 정확히 찾기 위한 기준 문서입니다. 실제 코드를 우선 기준으로 삼고, 이 문서는 작업 방향과 주요 연결 관계를 보조합니다.

## 1. 프로젝트 정체성

BGMS는 PUBG 유저를 위한 전술 지도, 전적 검색, 텔레메트리 리플레이, 전술 분석, AI 코칭을 제공하는 Next.js 애플리케이션입니다.

핵심 사용자 흐름은 다음과 같습니다.

```text
루트 접속
-> /maps/erangel 기본 리다이렉트
-> 지도/마커/핫드랍/제보/시뮬레이터 사용
-> 전적 검색 시 PUBG API 조회
-> 매치별 AnalysisEngine 분석
-> Supabase 캐시 및 스토리지 저장
-> 통계 UI, 지도 리플레이, AI 요약/코칭에서 재사용
```

## 2. 현재 기술 스택

- 프레임워크: Next.js 16.1.1, App Router
- UI: React 19.2.3, React DOM 19.2.3
- 지도: Leaflet, React Leaflet, leaflet.heat
- 스타일: Tailwind CSS v4, 전역 CSS 변수 기반 테마
- 아이콘: Lucide React
- 백엔드/인프라: Supabase Database, Auth, Storage, Realtime
- AI: AI SDK, Google Generative AI
- 모바일: Capacitor v8
- 테스트: Jest, Vitest

AI 모델 fallback 풀은 `app/api/pubg/ai-analyze/route.ts`와 `app/api/pubg/ai-summary/route.ts`의 실제 구현을 우선 확인합니다. 현재 중요한 fallback 예시는 아래와 같습니다.

```ts
const modelsToTry = [
  "gemini-3.1-flash-lite-preview",
  "gemini-3-flash-preview",
  "gemini-2.5-flash",
];
```

모델명 변경은 비용, 응답 형식, 스트리밍 안정성에 영향을 주므로 임의 변경하지 않습니다.

## 3. 앱 라우팅과 화면 흐름

### 루트와 지도

- `app/page.tsx`: 레거시 `?tab=` 쿼리를 새 경로로 리다이렉트하고, 기본 접속은 `/maps/erangel`로 보냅니다.
- `app/maps/[mapId]/page.tsx`: 맵 ID를 받아 `HomeClient`에 전달합니다.
- `app/HomeClient.tsx`: 지도 앱의 클라이언트 진입점입니다.
- `components/map/MapShell.tsx`: 지도 화면의 중심 컨테이너입니다. 사이드바, 도구 메뉴, 시뮬레이터, 텔레메트리 재생 상태를 조합합니다.
- `components/map/MapView.tsx`: Leaflet 지도, 타일, 마커, 그리드, 핫드랍, 리포트 위치, 텔레메트리 레이어를 실제 렌더링합니다.

### 전적/분석

- `app/stats/page.tsx`: 전적 검색 진입점입니다.
- `app/stats/[platform]/[nickname]/page.tsx`: 플랫폼과 닉네임을 URL로 받은 전적 분석 페이지입니다.
- `components/StatSearch.tsx`: 전적 검색과 매치 목록, 분석 UI의 주요 클라이언트 컴포넌트입니다.
- `components/stat/*`: 매치 카드, AI 요약, 레이더/스파이더 차트, 타임라인 등 분석 결과 표시 컴포넌트입니다.

### 커뮤니티/관리/도구

- `app/board/*`: 커뮤니티 게시판 목록, 상세, 작성 화면입니다.
- `app/admin/*`: 관리자용 리뷰, 맵 설정, 게임 데이터 관리 화면입니다.
- `app/map-editor/page.tsx`: 맵 마커 및 카테고리 편집 도구입니다.
- `app/backpack/*`, `app/weapons/*`: 아이템/무기 정보성 도구 화면입니다.

## 4. 핵심 데이터 파이프라인

### 플레이어 검색

`app/api/pubg/player/route.ts`

1. PUBG API에서 닉네임으로 accountId를 조회합니다.
2. 사용 가능한 시즌 목록과 최근 매치 ID를 가져옵니다.
3. 랭크/일반 시즌 통계를 함께 조회합니다.
4. FPP/TPP는 `roundsPlayed`가 많은 쪽을 대표 모드로 선택합니다.
5. 네트워크 오류는 `withRetry`로 재시도합니다.

### 매치 분석

`app/api/pubg/match/route.ts`

1. `matchId`, `nickname`, `platform`을 받아 PUBG Match API를 조회합니다.
2. 참가자, 로스터, 내 팀 정보를 계산합니다.
3. `processed_match_telemetry`에서 분석 결과 캐시를 확인합니다.
4. 캐시가 없거나 버전이 낮으면 텔레메트리를 가져옵니다.
5. 텔레메트리는 분석에 필요한 이벤트와 필드만 남겨 Supabase Storage에 저장합니다.
6. `AnalysisEngine`을 실행해 전술 지표를 계산합니다.
7. 대용량 지도 리플레이 데이터(`mapData`)는 Storage에 저장하고, 요약/전술 결과는 DB에 저장합니다.
8. `/api/pubg/ingest`를 백그라운드로 호출해 벤치마크 데이터 축적을 트리거합니다.

버전 관리는 `lib/pubg-analysis/constants.ts`의 값을 기준으로 합니다.

- `RESULT_VERSION`: 전술 분석 결과 캐시 무효화 기준
- `TELEMETRY_VERSION`: 리플레이/슬림 텔레메트리 캐시 무효화 기준

### 텔레메트리 리플레이

`app/api/pubg/telemetry/route.ts`와 `hooks/useTelemetry.ts`

1. 지도에서 `?playback={matchId}&nickname={nickname}` 형태로 리플레이를 요청합니다.
2. `useTelemetry`가 `/api/pubg/telemetry`를 호출합니다.
3. Storage에 저장된 `{matchId}_{nickname}_v{TELEMETRY_VERSION}_map.json` 캐시를 우선 사용합니다.
4. 캐시가 없으면 PUBG 텔레메트리를 직접 받아 `AnalysisEngine`을 실행하고 `mapData`를 생성합니다.
5. `TelemetryCanvasLayer`, `PlayerPathRenderer`, `PlayerMarkerRenderer`, `CombatRenderer`, `ShotRenderer`, `ZoneRenderer`가 지도 위에 재생 데이터를 렌더링합니다.

리플레이 성능은 이벤트 수, Canvas 렌더링, 상태 보간 로직에 크게 좌우됩니다. 대량 데이터 변경 시 `useTelemetry`와 `components/map/telemetry/*`를 함께 확인합니다.

### AI 코칭/요약

- `app/api/pubg/ai-analyze/route.ts`: 단일 매치 기반 코칭 리포트입니다.
- `app/api/pubg/ai-summary/route.ts`: 여러 매치 집계 기반 최근 플레이 요약입니다.

두 API 모두 분석 결과를 사람이 읽기 좋은 프롬프트 데이터로 변환한 뒤 Gemini 계열 모델에 전달합니다. 응답은 JSON 또는 NDJSON 스트리밍 형태로 소비될 수 있으므로, 프롬프트 구조와 클라이언트 파싱 로직을 함께 검토해야 합니다.

## 5. 주요 폴더 역할

### `app/`

Next.js App Router의 라우트와 API가 모여 있습니다.

- `layout.tsx`: 전역 레이아웃, 메타데이터, AuthProvider 래핑
- `globals.css`: Tailwind v4 및 전역 테마 변수
- `actions/map-settings.ts`: 맵 설정 관련 Server Action
- `api/pubg/*`: PUBG API, 텔레메트리, AI 분석, 벤치마크 관련 백엔드
- `api/admin/*`: 관리자 승인/거절/게임 데이터 관리
- `api/cron/*`: 핫드랍, 패치노트 예약 작업

### `components/`

UI와 클라이언트 상호작용 컴포넌트가 모여 있습니다.

- `components/map/*`: 전술 지도, 도구, 제보, 시뮬레이터, 텔레메트리 재생
- `components/map/telemetry/*`: 리플레이 전용 Canvas/레이어 렌더러
- `components/stat/*`: 전적 분석 카드, 차트, AI 요약, 매치 타임라인
- `components/board/*`: 게시판 UI
- `components/admin/*`: 관리자 편집 UI
- `components/common/*`: 글로벌 헤더, 모바일 메뉴, 하단 내비게이션, 푸터

### `hooks/`

클라이언트 비즈니스 로직을 캡슐화합니다.

- `useTelemetry.ts`: 리플레이 데이터 요청, 재생 시간, 플레이어 상태 보간
- `useMapData.ts`: 맵별 마커/제보 데이터 관리
- `useMapSettings.ts`: 맵 카테고리/아이콘 설정 동기화
- `useRealtimeToast.ts`: Supabase Realtime 기반 알림

### `lib/`

공유 설정, 분석 엔진, 도메인 유틸리티가 모여 있습니다.

- `lib/pubg-analysis/AnalysisEngine.ts`: 전술 분석 엔진 진입점
- `lib/pubg-analysis/handlers/*`: 전투, 위치, 자기장, 유틸리티, 리플레이 처리
- `lib/pubg-analysis/constants.ts`: 분석/텔레메트리 버전, 맵/무기 상수, 임계값
- `lib/pubg-analysis/types.ts`: 분석 결과와 내부 상태 타입
- `lib/map_config.ts`: 맵 좌표, 타일, 아이콘, 좌표 변환 설정
- `lib/seo-config.ts`: 동적 SEO 설정
- `lib/supabase.ts`, `lib/supabase-middleware.ts`: Supabase 클라이언트/세션 보조

### `utils/supabase/`

Next.js 서버/클라이언트 환경에 맞춘 Supabase SSR 클라이언트 생성 유틸리티입니다.

## 6. 변경 작업 시 우선 확인할 것

### 분석 엔진 수정

`lib/pubg-analysis/*`를 수정할 때는 아래 호출부를 같이 확인합니다.

- `app/api/pubg/match/route.ts`
- `app/api/pubg/telemetry/route.ts`
- `app/api/pubg/ingest/route.ts`
- `app/api/pubg/ai-summary/route.ts`
- `app/api/pubg/ai-analyze/route.ts`
- `components/stat/*`
- `hooks/useTelemetry.ts`

분석 결과 타입, `mapData` 구조, 캐시 버전이 바뀌면 `RESULT_VERSION` 또는 `TELEMETRY_VERSION` 증가가 필요할 수 있습니다.

### 텔레메트리 필드

PUBG 텔레메트리 필드는 `.pubg-telemetry-guide.md`를 우선 참고합니다.

- `maker`, `dBNOMaker`: 기절/DBNO 원인 추적
- `killer`, `finisher`: 최종 사살/마무리 추적
- `reviver`: 부활/소생 추적
- `attacker`, `victim`: 피해/교전 이벤트 추적
- `dBNOId`: 기절과 확킬/트레이드 인과관계 연결

필드명을 단순 추측으로 바꾸지 않습니다. `LogPlayerKill`, `LogPlayerKillV2`, `LogPlayerMakeGroggy`, `LogPlayerTakeDamage`, `LogPlayerRevive`는 서로 의미가 다릅니다.

### 지도/좌표 수정

지도 좌표나 타일 처리는 `lib/map_config.ts`, `components/map/MapView.tsx`, `components/map/telemetry/*`를 함께 봅니다.

PUBG 원본 좌표, 이미지 픽셀 좌표, Leaflet 좌표가 섞여 있으므로 다음을 반드시 확인합니다.

- 맵 크기: 8192, 4096, 3072, 2048 계열
- 원본 텔레메트리 좌표의 cm 단위 변환
- 맵별 y축 보정
- 리플레이 좌표와 일반 마커 좌표의 기준 일치

### Supabase/보안 수정

관리자 기능은 클라이언트 화면의 가드만으로 끝내지 않습니다. `app/api/admin/*` 라우트에서 반드시 서버 측 세션과 `profiles.role === "admin"` 검증을 수행합니다.

Service Role Key는 서버 라우트에서만 사용합니다. 클라이언트 컴포넌트에 노출하지 않습니다.

### RSC/Client 분리

페이지는 가능한 Server Component로 유지하고, 브라우저 API, Leaflet, localStorage, 상태/이벤트가 필요한 부분만 Client Component로 분리합니다. 지도와 통계 UI는 클라이언트 로직이 많으므로 import 경계를 특히 조심합니다.

## 7. 코드 품질 규칙

- 함수 시그니처를 바꾸면 호출부를 `rg`로 모두 추적합니다.
- 사용하지 않는 import, 주석 처리된 코드, 의미 없는 `console.log`는 남기지 않습니다.
- 임시 더미 데이터는 가능하면 실제 Supabase/API 흐름으로 연결합니다.
- 대용량 텔레메트리 데이터는 DB 행에 직접 넣기보다 Storage 캐시를 우선 고려합니다.
- 프롬프트/AI 응답 JSON 구조를 바꾸면 클라이언트 파싱과 에러 fallback을 함께 수정합니다.
- 성능에 민감한 지도/리플레이 변경은 데스크톱과 모바일 레이아웃을 모두 확인합니다.

## 8. 자주 쓰는 탐색 명령

```bash
rg "AnalysisEngine"
rg "RESULT_VERSION|TELEMETRY_VERSION"
rg "modelsToTry"
rg "playback"
rg "processed_match_telemetry|match_master_telemetry"
rg "profile.*role|role.*admin"
```

이 문서와 실제 코드가 충돌하면 실제 코드가 우선입니다. 문서가 틀린 것을 발견하면 변경 작업과 함께 이 파일도 갱신합니다.
