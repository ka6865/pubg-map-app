# BGMS 프로젝트 컨텍스트 및 작업 가이드
*최종 업데이트: 2026-06-12 (단순/교차 검색 시 최근 업데이트 오갱신 버그 해결 및 history.pushState 적용)*

이 문서는 BGMS 프로젝트에서 AI와 개발자가 빠르게 전체 흐름을 이해하고, 변경 지점을 정확히 찾기 위한 기준 문서입니다. 실제 코드를 우선 기준으로 삼고, 이 문서는 작업 방향과 주요 연결 관계를 보조합니다.

## 1. 프로젝트 정체성

BGMS는 PUBG 유저를 위한 전술 지도, 전적 검색, 텔레메트리 리플레이, 전술 분석, AI 코칭, 관리자 운영 보조를 제공하는 Next.js 애플리케이션입니다.

핵심 사용자 흐름은 다음과 같습니다.

```text
루트 접속
-> /maps/erangel 기본 리다이렉트
-> 지도/마커/핫드랍/제보/시뮬레이터 사용
-> 전적 검색 시 PUBG API 조회
-> 매치별 AnalysisEngine 분석
-> Cloudflare R2 하이브리드 캐시 및 Supabase DB 저장
-> 통계 UI, 지도 리플레이, AI 요약/코칭에서 재사용
```

관리자 운영 흐름은 다음과 같습니다.

```text
/admin/bot 접속
-> 운영판/AI 비서/승인 패널 확인
-> 조회성 진단은 즉시 실행
-> 삭제/발행/대량 변경은 승인 요청만 생성
-> agent_runs, agent_steps, agent_approvals, agent_memories에 기록
-> GitHub Actions가 일일 점검 snapshot을 남김
```

## 2. 현재 기술 스택

- 프레임워크: Next.js 16.1.1, App Router
- UI: React 19.2.3, React DOM 19.2.3
- 지도/3D: Leaflet, React Leaflet, leaflet.heat, Three.js
- 스타일: Tailwind CSS v4, 전역 CSS 변수 기반 테마
- 백엔드/인프라: Supabase Database, Auth, Realtime, Cloudflare R2 (AWS S3 SDK)
- AI: AI SDK, Google Generative AI
- 모바일: Capacitor v8
- 테스트: Jest, Vitest, ESLint

AI 모델 fallback 풀은 `app/api/pubg/ai-analyze/route.ts`와 `app/api/pubg/ai-summary/route.ts`의 실제 구현을 우선 확인합니다. 모델명은 비용, 응답 형식, 스트리밍 안정성, 런타임 지원 여부에 직접 영향을 주므로 코드와 운영 환경을 확인하지 않고 임의로 변경하지 않습니다.

### 텔레메트리 리플레이

`app/api/pubg/telemetry/route.ts`와 `hooks/useTelemetry.ts`가 리플레이 데이터 흐름의 중심입니다.

1. 지도에서 `?playback={matchId}&nickname={nickname}` 형태로 리플레이를 요청합니다.
2. `useTelemetry`가 `/api/pubg/telemetry`를 호출합니다.
3. R2 스토리지에 저장된 `{matchId}_v{TELEMETRY_VERSION}_map_{mode}.json` 캐시를 우선 사용합니다.
4. 캐시가 없으면 PUBG 텔레메트리를 직접 받아 `AnalysisEngine`을 실행하고 `mapData`를 생성하여 R2에 업로드합니다.
5. `components/map/telemetry/*` 렌더러가 지도 위에 재생 데이터를 표시합니다.
6. 3D 리플레이는 대형 맵의 하이트맵, 고도 보정, 지상 스냅, 카메라/레이어 토글, 고해상도 타일 텍스처를 사용합니다.

3D 리플레이 관련 변경은 `app/replay/3d/page.tsx`, `lib/map_config.ts`, `hooks/useTelemetry.ts`, `components/map/telemetry/*`를 함께 확인합니다. 지형, 충돌, 고도, 카메라, 리플레이 ingest, fallback, 가시성 QA가 서로 연결되어 있습니다.

### AI 코칭/요약

- `app/api/pubg/ai-analyze/route.ts`: 단일 매치 기반 코칭 리포트입니다.
- `app/api/pubg/ai-summary/route.ts`: 여러 매치 집계 기반 최근 플레이 요약입니다.
- `app/api/pubg/ai-squad/route.ts`: 스쿼드 조합 기반 AI 코칭 리포트입니다.
- `lib/pubg-analysis/squadAnalysis.ts`: 스쿼드 그룹 탐지, 팀 지표 집계, 벤치마크 비교, 역할 프로필 산출, 원인 장면 후보 산출을 담당하는 공용 서비스입니다. `ai-squad`와 `squad-analyze` route가 이 서비스를 함께 사용하며, `squad-analyze` 상세 응답에는 `causeScenes`가 포함됩니다. 스쿼드 리포트의 `totalSmokeRescues`와 `totalRevives`는 저장된 플레이어별 `tradeStats.smokeRescues/revCount`가 아니라 `fullResult.timeline`의 팀 단위 연막/소생 이벤트에서 재산출하고, 타임라인이 없거나 복구 관련 이벤트 신호가 없는 구 데이터에서만 저장 지표로 fallback합니다.
- `lib/pubg-analysis/squadRecoveryStats.ts`: 스쿼드 리포트용 복구 지표 순수 로직입니다. `TEAM_KNOCK`/`DOWNED` 이후 15초 안의 연막 사용, 30초 안의 동일 대상 `REVIVE`/`TEAM_REVIVE`, 위치가 있으면 100 좌표 단위 이내 조건을 함께 만족할 때 연막 구출 후보로 계산합니다. `hasSquadRecoveryTimelineSignals`는 복구 관련 이벤트가 없는 부분 타임라인을 저장 지표 fallback 대상으로 분리합니다.
- `lib/pubg-analysis/squadCauseScenes.ts`: 스쿼드 AI 고도화 전 검증용 원인 장면 추출 순수 로직입니다. `timeline`, `tradeStats`, `isolationData`에서 `late_trade`, `isolation_death`, `no_smoke_rescue`, `revive_save`, `safe_revive_without_smoke`, `recall_recovery`, `clutch_recovery`, `team_collapse`, `focus_fire_success`, `team_wipe` 후보를 만들고 AI 프롬프트용 제한 컨텍스트를 생성합니다. 프롬프트는 `riskFlags`와 `unsupportedClaims`를 요구해 AI가 성향/의도/소생 시도 여부처럼 측정되지 않은 주장을 자가 점검하도록 제한합니다. `RECALL`, `TEAM_RECALL`, `REDEPLOY`는 전력 복구 성공 근거로만 사용하며, 별도 원본 파싱 없이 블루칩 회수자/회수 시점을 단정하지 않습니다. `maxScenes >= 5`일 때 성공 복구 장면이 있으면 최소 1개를 포함해 문제 장면만으로 AI 입력이 편향되지 않도록 선택합니다. 같은 매치/같은 원인 유형 또는 같은 대상자의 반복 장면은 장면이 부족한 경우가 아니면 뒤로 미룹니다. `TEAM_KNOCK`/`DOWNED`는 `기절 이벤트`, `TEAM_DIED`/`DIED`는 `사망 이벤트`로 facts 문구를 분리합니다. `reviveWithin30s: true`인 고립 지표 장면은 소생 성공 근거를 facts에 넣고 위험도를 한 단계 낮춥니다. 고립 위험 장면은 사용자에게 `고립지수`를 먼저 노출하지 않고 `팀과 N미터 떨어진 상태`, `위험 기준 N미터`, `팀 간격 위험도` 순서로 설명합니다. `validateSquadCauseSceneAiText`는 AI 응답 후처리에서 `지원 부족`, `팀 응집력`, `복구 능력`, `전투력`, `전투/교전 진입 각`, `소생 시도/성공 강요`, `소생/복구 성공률`, `즉각적인 연막`, `즉각적인 공격/복구/백업`, `반드시`, `불가능`, `안전성 단정`, `엄폐물/시야 단정`, `교전 회피 단정`, `전력 손실 인과 단정`, `전력 복구 실패`, `적 전원 처치 확인`, `팀 전멸 주체 모호성`, `침착하게` 같은 미측정 단정 표현을 경고로 분리하며, `소생 성공 이벤트` 같은 근거 문장과 "적 전원 처치 확정 장면이 아님/아니므로" 같은 부정 문맥 및 `적 전멸 기여` 문맥은 오탐하지 않도록 처리합니다.
- `scripts/experiment_squad_cause_scenes.ts`: 원인 장면 AI 피드백 실험용 스크립트입니다. fixture 또는 Supabase `processed_match_telemetry`를 읽어 원인 장면과 프롬프트를 출력합니다. 실제 Gemini 호출은 `--ai` 옵션이며, 실제 유저 매치 데이터 외부 전송은 사용자의 명시적 승인 후에만 실행합니다. `--from-db` 조회는 `--platform` 기본값 `steam`과 정규화된 `player_id`를 함께 조건으로 사용하고, 내부 `fullResult.stats.name/platform`이 요청값과 다르면 캐시를 무시합니다. `--ai` 실행 시 기본 모델 실패 또는 503/429 같은 일시 오류가 발생하면 `--ai-retries`와 `--fallback-models` 기반으로 재시도하고, 장면 추출 성공과 AI 응답 실패를 구분해 출력합니다.

세 API 모두 분석 결과를 사람이 읽기 좋은 프롬프트 데이터로 변환한 뒤 Gemini 계열 모델에 전달합니다. 응답은 JSON 또는 NDJSON 스트리밍 형태로 소비될 수 있으므로, 프롬프트 구조와 클라이언트 파싱 로직을 함께 검토해야 합니다.

원인 장면 레이어는 `squad-analyze` 상세 응답의 `causeScenes`와 `components/stat/SquadCauseScenes.tsx`를 통해 `SquadAnalysisPanel`에 1차 운영 연결되었습니다. 현재 UI 연결은 Gemini 추가 호출 없이 서버가 계산한 `facts`, `metrics`, `severity`, `confidence`, `replayWindow`를 모바일 대응 카드로 보여주며, 모바일에서는 기본 접힘 상태로 노출하고 `sm` 이상 화면에서는 기본 펼침 상태로 렌더링합니다. 카드의 `2D 맵에서 보기`는 기존 스쿼드 2D 맵 섹션을 펼치고 해당 `matchId`를 선택합니다.

전적 분석 저장 identity 표준은 `match_id + platform + player_id`입니다. `app/api/pubg/match/route.ts`와 `app/api/pubg/ingest/route.ts`는 분석 대상 유저 1명만 `processed_match_telemetry`에 저장하며, 더 이상 같은 분석 결과를 팀원 전체에 복사 저장하지 않습니다. `lib/pubg-analysis/cacheIdentity.ts`는 읽기 경로에서 `fullResult.stats.name`, `fullResult.player_id`, `fullResult.platform`을 요청 `player_id/platform`과 대조해 오염 row를 무시합니다. 기존 오염 row는 자동 삭제하지 않고, `scripts/audit_processed_telemetry_identity.ts --dry-run`으로 샘플과 개수를 확인한 뒤 `--apply`가 있을 때만 삭제합니다.

2026-06-12 운영 DB에는 `20260612011506_safe_platform_identity_backfill`와 `20260612011615_drop_legacy_match_stats_raw_unique`가 적용되었습니다. `processed_match_telemetry`, `match_stats_raw`, `global_benchmarks`는 복구 가능한 기존 row를 `data.fullResult.platform` 또는 같은 `match_id`의 단일 platform 신호로 `steam/kakao` 백필하고, 플랫폼을 확정할 수 없는 과거 `global_benchmarks` row는 `legacy_unknown`으로 격리합니다. `legacy_unknown`은 신규 전적 검색 조회 대상이 아니라 과거 보존/감사용 값입니다. 적용 직후 운영 분포는 `processed_match_telemetry` steam 496건/kakao 53건, `match_stats_raw` steam 10,584건/kakao 1,941건, `global_benchmarks` steam 111건/kakao 16건/legacy_unknown 2,348건입니다. `/api/pubg/battle`은 `platform1/platform2` 기준으로 `global_benchmarks`를 조회하며, 플랫폼이 URL에 없으면 `pubg_player_cache`에서 단일 플랫폼일 때만 보정하고 다중 후보 또는 확인 불가 상태는 명시 선택을 요구합니다.

개인 행동 지표는 분석 대상 플레이어 기준으로만 저장합니다. `itemUseSummary.smokes/frags/molotovs/others`, `itemUseStats.heals/boosts/throwCount/lethalThrowCount`, `global_benchmarks.smoke_count/frag_count/lethal_throw_count`는 팀원 행동을 합산하지 않습니다. 팀원 연막, 회복, 부스트 사용은 `timeline`의 `ITEM_USE`에는 남아 스쿼드 리포트 재계산에 쓰일 수 있지만 개인 지표와 벤치마크에는 저장하지 않습니다. 이 수정 전 생성된 일부 `global_benchmarks.smoke_count`와 `itemUseSummary.smokes`는 팀원 연막이 섞였을 수 있으나, 집계 row만으로 안전한 자동 보정은 불가능하므로 신규 분석 차단과 재분석 시 자연 교체 정책을 사용합니다.

Admin Agent 데이터 품질 관리는 `lib/admin-agent/data-quality.ts`를 기준으로 합니다. `app/api/admin/agent/monitor/route.ts`는 최근 `DATA_QUALITY_AUDIT_RECENT_DAYS`일, 최대 `DATA_QUALITY_AUDIT_MAX_ROWS` row를 dry-run으로 감사하고, identity mismatch가 있으면 `repair_processed_telemetry_identity` 승인 요청을 생성합니다. 승인 요청은 중복 pending이 있으면 새로 만들지 않으며, 실행 시에도 targets를 다시 조회해 현재도 mismatch인 row만 삭제합니다. `scripts/audit_processed_telemetry_identity.ts`는 같은 helper를 사용하는 로컬/GitHub Actions용 감사 스크립트입니다. `.github/workflows/daily-tasks.yml`은 매일 삭제 없는 dry-run 감사를 실행하고, 실제 삭제는 `/admin/bot` 승인 패널에서 impact 확인과 `confirmedImpact=true`를 거친 뒤에만 실행됩니다.

Analytics 운영 데이터는 `analytics_events`에 저장됩니다. `lib/analytics.ts`는 기본적으로 개발 모드, `localhost`, `127.0.0.1`, `.local` host의 Supabase mirror 저장을 차단하며, 로컬 저장이 꼭 필요할 때만 `NEXT_PUBLIC_ANALYTICS_MIRROR_LOCAL=true`로 명시 허용합니다. 서버 라우트 `app/api/analytics/event/route.ts`도 `ANALYTICS_ACCEPT_LOCAL=true`가 없으면 로컬 host 요청을 저장하지 않습니다. 저장되는 이벤트에는 `client_environment`, `source_host`, `is_internal` 원천 정보가 포함됩니다. Admin Agent의 `buildUserMetricsSummary`는 전적 검색 대상 닉네임과 로그인 활동 회원을 분리해, `/stats/steam/KangHeeSung_` 조회수를 KangHeeSung_ 계정의 활동으로 단정하지 않습니다.

저장소 용량 관제는 `lib/admin-agent/storage-health.ts`를 기준으로 합니다. Supabase DB는 `get_db_size` RPC 기준으로 Free DB 500MB 한도 대비 사용률을 계산하고, Cloudflare R2는 `getR2BucketUsage`가 `ListObjectsV2` pagination을 돌며 객체 수와 총 바이트를 집계합니다. `/api/admin/dashboard`는 기존 `r2Cache` 응답을 유지하면서 `storageHealth` 전체를 내려주고, `inspect_operations`의 `cache_health`도 같은 공용 helper를 사용합니다. 2026-06-11 점검 기준 운영 DB는 약 120.6MB, R2는 약 645.3MB로 둘 다 안정권입니다.

Codex 상시 작업 규칙은 루트 `AGENTS.md`를 기준으로 합니다. `.codex-drafts/`는 gitignore 대상인 검토용 초안 폴더이며, 하네스 설계 기록과 Skill 원본 후보를 둡니다. 2026-06-12 기준 `bgms-pubg-analysis`, `bgms-supabase-ops`, `bgms-admin-agent`, `bgms-frontend-qa`는 `~/.codex/skills/bgms-*`에 실제 설치되었습니다. 설치본은 다음 새 대화부터 Codex Skill 후보로 자동 발견될 수 있습니다. Smoke test나 실제 작업에서 파일 범위, 검증 명령, 금지 조건이 부정확하다고 확인되면 `.codex-drafts/skills/*`와 `~/.codex/skills/bgms-*` 설치본을 함께 갱신합니다. 방향성, 고도화 계획, 개선안, 설계 검토 요청은 구현 전에 현재 문서와 실제 코드를 확인하고 목표, 현재 구조, 단계, 영향, 얻는 점/잃는 점, 위험, 검증 방법을 먼저 제시합니다. Context7/Tavily MCP 설정은 프로젝트 코드가 아니라 로컬 Codex 운영 자산이며 `~/.codex/config.toml`에서 관리합니다. MCP smoke test는 개인 API 키, 네트워크, 새 세션 로딩 상태에 의존하므로 `package.json`의 `verify:*` 하네스에는 포함하지 않습니다.

벤치마크 조회는 `lib/pubg-analysis/benchmarkLookup.ts`를 통해 `game_mode + match_type + tier` exact 조회를 먼저 수행하고, exact row가 없으면 같은 티어군을 `match_count/sample_count` 기반 가중 평균으로 fallback합니다. `/api/pubg/battle` 평균 티어는 `lib/pubg-analysis/tierAveraging.ts`에서 실제 `score` 평균을 우선 사용하고, score가 없을 때만 문자 티어 fallback을 사용합니다.

AI 캐시는 `lib/pubg-analysis/constants.ts`의 `AI_CACHE_VERSION`을 `prompt_version`으로 저장합니다. 프롬프트 또는 응답 구조가 바뀌면 이 상수를 변경해 기존 캐시와 새 응답을 분리합니다. 현재 캐시 고유키는 다음과 같습니다.

- `match_ai_coaching_cache`: `match_id + platform + player_id + coaching_style + prompt_version`
- `player_ai_summary_cache`: `player_id + platform + match_ids_hash + prompt_version`
- `squad_ai_coaching_cache`: `player_id + platform + group_key + match_ids_hash + coaching_style + prompt_version`

AI 캐시 테이블은 RLS를 활성화하고 클라이언트 직접 접근 정책을 두지 않는 서버/service role 경유 구조입니다. `ai-summary`는 `force=true`일 때 AI 결과 캐시 조회를 건너뛰고 새 Gemini 결과를 생성해 `upsert`합니다. `ai-summary`와 `ai-squad`의 서버 모델 재시도 총 제한은 약 22초이며, `RecentAISummary` 클라이언트 안전 타임아웃은 35초입니다.

## 4. PUBG API 호출 정책

PUBG API는 운영상 `1분 10회` 제한을 기준으로 보호합니다. 검색 체감 속도보다 호출 안정성이 더 중요합니다.

- 기본 전적 검색은 플레이어/시즌/매치 중심이며, `refresh=true`가 없을 경우 무조건 Supabase DB 캐시(`pubg_player_cache`)에 저장된 통계(`season_stats_data`), 최근 매치 목록(`recent_match_ids`) 등을 활용해 즉시 리턴함으로써 외부 API 호출을 원천 차단하고 쿨다운 및 속도를 보장합니다.
- URL의 `_t` 타임스탬프 파라미터는 강제 갱신 조건에서 제외하여 단순 페이지 이동 시 불필요한 API 직접 호출을 차단합니다.
- PUBG API를 직접 호출하여 데이터 갱신 시, DB에 `season_stats_data`, `last_season_id`, `recent_match_ids`, `seasons_list` 데이터를 온전하게 갱신 저장하고 응답 바디에도 동일한 생성 시각(`updatedAt`)을 명시하여 캐시 히트와 직접 호출 간의 시각 동기화 오류를 해결합니다.
- 수동 전적 갱신은 `refresh=true` 쿼리가 전송될 때만 PUBG API를 호출하며, 갱신 성공 시 1분간의 쿨타임(버튼 비활성화)이 적용됩니다.
- 기본 전적 검색에서는 무기 숙련도 API를 호출하지 않습니다.
- 무기 숙련도는 `/api/pubg/player/weapon-mastery`에서만 갱신합니다.
- 무기 숙련도 갱신은 `pubg_player_cache.id`의 accountId를 사용합니다.
- accountId 캐시가 없으면 PUBG player lookup을 추가 호출하지 않고 `409`로 "전적 검색 먼저 필요"를 반환합니다.
- 최근 3시간 이내 `mastery_updated_at`이 있으면 PUBG API를 다시 호출하지 않고 DB 캐시를 반환합니다.
- `trackPubgRateLimit`와 `pubg_api_status`로 API 잔여량을 기록합니다.
- PUBG API 에러는 `reportPubgApiError`를 통해 운영 진단에 연결합니다.

무기 마스터리 파싱과 카테고리 분류는 `lib/pubg/weaponMastery.ts`를 우선 확인합니다. 카테고리는 전체, AR, SMG, DMR, SR, 샷건, LMG, 권총, 투척류, 근접/특수, 기타 기준으로 정리합니다.

## 8. Admin Agent 운영 원칙

Admin Agent는 자동 실행 봇이 아니라, 무료 플랜에서도 안전하게 굴러가는 운영 감시/승인 보조 시스템입니다.

- 서버에서도 admin role을 검증합니다.
- `/api/admin/system` 같은 캐시 삭제 API도 `withAuthGuard`와 admin role 검증을 통과해야 실행됩니다.
- Service Role Key는 서버 라우트 안에서만 사용합니다.
- 조회성 진단은 바로 실행할 수 있습니다.
- 게시글 발행, 캐시 삭제, 유저 조작, 대량 수정 같은 위험 작업은 승인 요청만 생성합니다.
- 승인 요청에는 영향도 미리보기를 붙입니다.
- 승인 실행 직전에도 영향도를 다시 계산하고 결과 로그에 포함합니다.
- 모든 실행은 `agent_runs`, `agent_steps`, `agent_approvals`, `agent_memories` 기준으로 추적합니다.
- Discord 알림은 위험 조건이 있을 때만 보냅니다.
- `DISCORD_WEBHOOK_URL`이 없으면 알림은 조용히 건너뜁니다.

주요 코드 위치는 다음과 같습니다.

- `lib/admin-agent/tools.ts`: Agent가 사용할 수 있는 도구 정의와 안전 등급
- `lib/admin-agent/approvals.ts`: 승인 요청 생성/조회
- `lib/admin-agent/approval-execution.ts`: 승인된 작업 실행
- `lib/admin-agent/impact.ts`: 승인 영향도 계산
- `lib/admin-agent/monitor-trend.ts`: 운영 snapshot 추세 계산
- `lib/admin-agent/thresholds.ts`: 비용, 에러, PUBG API 잔여량 등 임계치
- `lib/admin-agent/logging.ts`: 실행 로그 기록

## 9. 관리자 하루 운영 루틴

관리자는 코드를 몰라도 `/admin/bot` 기준으로 하루 운영 상태를 확인할 수 있어야 합니다.

### 하루 시작

1. `/admin/bot`에서 운영판을 먼저 봅니다.
2. 승인 대기 수, PUBG API 에러, API 잔여량, AI 비용, 배포 상태를 확인합니다.
3. `수동 점검`, `운영판 보기`, `최종 보고` 계열 버튼으로 Agent snapshot을 갱신합니다.
4. 승인 대기 작업이 있으면 impact summary를 보고 승인/거절/보류를 결정합니다.
5. 위험 작업은 채팅으로 바로 실행하지 않고 승인 큐를 거칩니다.

### 문제 발생 시

1. monitor alert의 severity와 message를 먼저 확인합니다.
2. PUBG API 문제면 `pubg_api_status`, `pubg_api_errors`, 최근 검색/매치 분석 API를 같이 봅니다.
3. AI 비용 문제면 `ai_usage_logs`와 AI 요약/코칭 호출부를 확인합니다.
4. 배포 문제면 GitHub Actions, Vercel deployment, `/api/admin/agent/rollout`을 같이 봅니다.
5. 캐시/텔레메트리 문제면 R2 상태, `processed_match_telemetry`, `TELEMETRY_VERSION`을 확인합니다.
6. AI 캐시 문제면 `AI_CACHE_VERSION`, 세 AI 캐시 테이블의 `prompt_version`, `platform`, `player_id`, `match_ids_hash` 조건을 확인합니다.

### 하루 마감

1. Daily Checkout 또는 최종 보고를 실행합니다.
2. 남은 승인 대기, monitor alert, rollout readiness fail을 정리합니다.
3. 해결하지 못한 이슈는 handoff/incident/memory로 남깁니다.
4. 반복 이슈는 `agent_memories`에 저장해 다음 대응에 재사용합니다.

## 10. 자동화와 무료 플랜 운영

Vercel 무료 플랜을 고려해 별도 Vercel Cron을 늘리지 않습니다. 긴 작업은 기존 GitHub Actions에 남기고, Agent는 관찰, 기록, 알림, 승인 영향 분석을 담당합니다.

`.github/workflows/daily-tasks.yml`의 주요 역할은 다음과 같습니다.

- R2 스토리지 상태 점검
- 오래된 텔레메트리/AI 캐시 정리
- 스마트 스크래퍼 실행
- 블루존 데이터 추출
- 패치노트 동기화
- Agent Monitor Snapshot 기록
- Agent Rollout Readiness Snapshot 기록

`/api/admin/agent/monitor`는 `severity`, `alerts`, `recommendations`를 반환합니다. 정상 상태는 `agent_runs`에만 기록하고 Discord 알림은 보내지 않습니다. 위험 조건이 있을 때만 짧은 운영 알림을 보냅니다.

## 12. 변경 작업 시 우선 확인할 것

### 분석 엔진 수정

`lib/pubg-analysis/*`를 수정할 때는 아래 호출부를 같이 확인합니다.

- `app/api/pubg/match/route.ts`
- `app/api/pubg/telemetry/route.ts`
- `app/api/pubg/ingest/route.ts`
- `app/api/pubg/ai-summary/route.ts`
- `app/api/pubg/ai-analyze/route.ts`
- `components/stat/*`
- `hooks/useTelemetry.ts`

분석 결과 타입, `mapData` 구조, 캐시 버전이 바뀌면 `RESULT_VERSION` 또는 `TELEMETRY_VERSION` 증가가 필요할 수 있습니다.

### R2 스토리지 및 클린업 규칙

- 대용량 텔레메트리 가공 원본 데이터는 Supabase DB에 적재하지 않고, Cloudflare R2 버킷 `bgms`에 저장합니다.
- R2 객체 수/용량 집계는 `getR2BucketUsage`로 페이지네이션 전체 집계를 사용합니다. `listR2Files(limit)`는 파일 목록이 실제로 필요한 제한 조회에만 사용합니다.
- `scripts/cleanup_telemetry.ts` 스크립트를 사용해 고립 파일을 지울 때는 R2 API의 벌크 삭제(`deleteMultipleFromR2`)를 사용합니다.
- R2 서비스 임포트 시 Next.js 절대 경로(`@/`)를 사용할 수 없는 단독 실행 스크립트에서는 상대 경로(`../lib/pubg-analysis/r2Service`)를 유지합니다.

### 텔레메트리 필드

PUBG 텔레메트리 필드는 `.pubg-telemetry-guide.md`를 우선 참고합니다.

- `maker`, `dBNOMaker`: 기절/DBNO 원인 추적
- `killer`, `finisher`: 최종 사살/마무리 추적
- `reviver`: 부활/소생 추적
- `attacker`, `victim`: 피해/교전 이벤트 추적
- `dBNOId`: 기절과 확킬/트레이드 인과관계 연결

필드명을 단순 추측으로 바꾸지 않습니다. `LogPlayerKillV2`, `LogPlayerMakeGroggy`, `LogPlayerTakeDamage`, `LogPlayerRevive`는 서로 의미가 다릅니다.

### 지도/좌표 수정

지도 좌표나 타일 처리는 `lib/map_config.ts`, `components/map/MapView.tsx`, `components/map/telemetry/*`를 함께 봅니다.

- 맵 크기: 8192, 4096, 3072, 2048 계열
- 원본 텔레메트리 좌표의 cm 단위 변환
- 맵별 y축 보정
- 리플레이 좌표와 일반 마커 좌표의 기준 일치

### Supabase/보안 수정

관리자 기능은 클라이언트 화면의 가드만으로 끝내지 않습니다. `app/api/admin/*` 라우트에서 반드시 서버 측 세션과 `profiles.role === "admin"` 검증을 수행합니다.

Service Role Key는 서버 라우트에서만 사용합니다. 클라이언트 컴포넌트에 노출하지 않습니다.

### RSC/Client 분리

페이지는 가능한 Server Component로 유지하고, 브라우저 API, Leaflet, localStorage, 상태/이벤트가 필요한 부분만 Client Component로 분리합니다. 지도와 통계 UI는 클라이언트 로직이 많으므로 import 경계를 특히 조심합니다.

## 15. 테스트와 점검 루틴

변경 범위가 작아도 타입 검사와 관련 테스트는 먼저 돌립니다.

```bash
npm run verify:core
npm run verify:analysis
npm run verify:admin
```

`verify:*` 하네스는 Codex/작업 루프용 빠른 검증 기준입니다.

- `verify:core`: `eslint`와 `tsc --noEmit --pretty false`
- `verify:analysis`: 전적 분석, AI 캐시, 스쿼드 원인 장면, 복구 지표 관련 Vitest
- `verify:admin`: Admin Agent, data quality, analytics/user metrics 관련 Vitest

모바일 사용자는 BGMS의 핵심 사용자이므로 화면에 보이는 기능 구현, 업데이트, 문구/카드/버튼/탭/필터 변경은 모바일 UI 확인을 기본 검증에 포함합니다. 최소 모바일 뷰포트는 `375x667`, `390x844`, `430x932`로 두고, 필요 시 `768x1024`, `1280x720`을 추가합니다. 확인 항목은 텍스트 겹침, 터치 영역, 탭/필터 줄바꿈, 기본 접힘 상태, 로딩/빈/에러 상태, 하단 내비게이션 간섭입니다. 데스크톱 전용 변경처럼 보여도 공통 컴포넌트나 전역 CSS를 건드렸다면 모바일 확인을 생략하지 않습니다.

로컬 UI와 모바일 반응형 QA는 Browser/in-app Browser를 우선 사용합니다. Chrome은 실제 사용자 Chrome 프로필, 로그인 쿠키, 확장 프로그램, 기존 탭 상태가 필요한 경우에만 사용합니다. Chrome 연결이 불안정하면 로컬 서버 상태, dev server 첫 컴파일, 확장 연결 상태를 확인하고 Browser로 대체 검증할 수 있습니다. Codex의 모바일 디버깅은 웹 모바일 뷰포트와 반응형 레이아웃 확인에 유효하지만, iOS Safari 고유 버그, Android WebView, Capacitor 네이티브 런타임, 실제 기기 권한/푸시/키보드/주소창 변화는 실기기 또는 에뮬레이터 확인이 필요할 수 있습니다.

Codex 작업 셸에서 `npm`, `node`, `npx`가 PATH에 없을 수 있으므로, 프로젝트 검증은 `/opt/homebrew/bin` 또는 `/usr/local/bin`의 일반 Node/npm을 우선 사용합니다. Codex 번들 Node/npm은 MCP 실행에는 적합하지만, Vitest/Rolldown 같은 네이티브 바인딩 검증에서는 macOS 코드 서명 문제로 실패할 수 있으므로 기본 검증 Node로 단정하지 않습니다. `verify:*` 스크립트는 환경 차이를 줄이기 위해 중첩 `npm run ...` 호출을 피합니다.

로컬 UI QA가 실패하면 실제 UI 버그로 단정하기 전에 런타임 PATH, 기존 3000번 dev server, `.next/dev/lock`, `localhost`와 `127.0.0.1` 접근 차이, Browser 도구 노출 여부를 순서대로 확인합니다. Browser/in-app Browser가 없어서 Puppeteer나 Chrome으로 우회할 때는 모바일/데스크톱 검증을 분리하고, 텍스트 추측 selector보다 `data-testid` 같은 안정적 selector를 우선합니다. 최종 보고에서는 서버 접근 실패, selector 실패, 로딩 타임아웃, 실제 UI 깨짐을 분리해 기록합니다.

필요 시 더 넓게 아래 개별 테스트를 추가합니다.

```bash
npx vitest run tests/admin-agent-api.test.ts tests/admin-bot.test.ts tests/security.test.ts
npx vitest run tests/analysis-engine.test.ts tests/squad-analysis.test.ts
npx vitest run tests/squad-cause-scenes.test.ts tests/squad-recovery-stats.test.ts

```

테스트 폴더 기준은 다음과 같습니다.

- `tests/**/*.test.ts`: Vitest 기반 서버/API/순수 로직 테스트입니다.
- `__tests__/**/*.test.tsx`: Jest/jsdom 기반 UI 또는 브라우저 환경 테스트입니다.

## 16. 유지보수 기본 원칙

처음 보는 개발자나 AI는 아래 순서로 접근합니다.

1. 사용자가 보는 화면 경로를 먼저 찾습니다.
2. 화면에서 호출하는 API route를 찾습니다.
3. API route가 읽고 쓰는 Supabase 테이블과 R2 키를 확인합니다.
4. 분석/도메인 로직은 `lib/*`에서 찾습니다.
5. 캐시가 있으면 메모리 캐시, DB 캐시, R2 캐시, ISR/태그 캐시를 구분합니다.
6. 관리자 기능이면 서버 측 admin 검증과 Service Role Key 노출 여부를 먼저 점검합니다.
7. PUBG API를 추가 호출하는 변경은 rate limit 영향부터 계산합니다.
8. 위험 작업은 직접 실행 로직보다 승인 큐 흐름에 태우는 것을 기본값으로 둡니다.

## 17. 코드 품질 규칙

- 위험한 운영 액션은 자동 실행하지 말고 Admin Agent 승인 흐름을 사용합니다.
