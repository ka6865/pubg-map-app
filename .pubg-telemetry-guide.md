# BGMS PUBG Telemetry Tactical Analysis Guide (V11.9.2)

이 문서는 BGMS의 PUBG 매치/텔레메트리 수집, 전술 지표 계산, AI 전적 분석 흐름을 설명합니다. 기준은 PUBG 공식 Web API 및 Telemetry Events 문서이며, 공식 문서에 명시된 이벤트를 core source로 사용합니다.

Official reference:
- PUBG API Documentation: https://documentation.pubg.com/en/index.html
- Telemetry Events: https://documentation.pubg.com/en/telemetry-events.html
- Usage Guide / Caching: https://documentation.pubg.com/en/usage.html

---

## 🚨 [V11.9.2] 물리량 정규화 표준 (Normalization Standard)
*본 섹션은 V11.9.2 엔진 업데이트에 따라 추가되었습니다.*

백엔드 엔진(`processor.ts`)에서 수집 시점에 즉시 정규화를 수행하여 데이터 일관성을 유지합니다.

- **거리 및 높이 (Distance/Height)**: 
  - 원본 cm 좌표를 즉시 1/100 처리하여 **미터(m)** 단위로 통일.
  - *주의*: `processor.ts` 이후의 모든 레이어(API, AI, UI)는 추가적인 `/ 100` 연산 없이 데이터를 그대로 사용함.
- **자기장 피해 (Bluezone Waste)**: 
  - 생존 시간을 기반으로 한 추정치가 아닌, `LogPlayerTakeDamage`에서 발생하는 실제 **HP(데미지 총량)**를 누적 집계함.
- **반응 속도 (Latency)**: 
  - 밀리초(ms) 데이터를 **초(s)** 단위(예: 0.69s)로 변환하여 AI 및 UI에 전달함.

## 1. Data Pipeline

### Match Analysis API: `/api/pubg/match`
- PUBG Match API에서 매치 상세, 참가자, 로스터, 텔레메트리 asset URL을 가져옵니다.
- 원본 텔레메트리를 분석에 필요한 공식 이벤트 중심으로 slimming합니다.
- 분석용 캐시는 Supabase Storage의 `${matchId}_analyze.json`에 저장합니다.
- `match_master_telemetry.storage_path`가 분석 캐시 경로와 일치하고 파일이 실제로 존재하며 비어 있지 않을 때만 재사용합니다.
- 최종 분석 결과는 `processed_match_telemetry`에 저장합니다.

### Map Replay API: `/api/pubg/telemetry`
- 지도 리플레이용 이벤트를 0-8192 맵 좌표로 정규화합니다.
- 지도 캐시는 `matchId + accountId + mode` 기준으로 분리합니다.
- `mode=full`은 적 위치 전량, `mode=lite`는 적 위치 샘플링을 적용합니다.
- 지도 캐시는 분석 캐시 버전으로 간주하지 않습니다.

### AI APIs
- `/api/pubg/ai-analyze`: 단일 매치의 전술 지표를 Gemini에 전달하여 코칭 리포트를 생성합니다.
- `/api/pubg/ai-summary`: 최근 10경기의 전술 지표를 집계해 토론형 요약 리포트를 생성합니다.
- AI에 전달하는 값은 가능한 한 `sampleCount`, `measured` 또는 "측정 불가" 상태를 함께 제공해야 합니다.
- **[V11.9.2]**: 10경기 요약 시 단일 루프(O(N)) 최적화를 적용하여 지표 누락을 방지하고, 모든 물리 지표를 정규화된 단위(m, HP, s)로 전달함.

## 2. Official Event Source Policy

BGMS 전술 분석은 공식 Telemetry Events에 명시된 이벤트를 core source로 사용합니다. 공식 문서에서 확인되지 않는 이벤트는 optional enrichment로만 사용하고, 지표의 필수 전제에 두지 않습니다.

### Core Events
- `LogMatchStart`: 매치 시작 시각 및 기본 메타데이터
- `LogPlayerCreate`: 초기 생존 상태
- `LogPlayerPosition`: 위치, 이동, 고립도, 거리 기반 지표
- `LogPlayerAttack`: 사격/공격 시도
- `LogPlayerUseThrowable`: 투척물 사용 시점, 사용자, 무기
- `LogPlayerTakeDamage`: 실제 피해, 피해 원인, 교전 연결
- `LogPlayerMakeGroggy`: DBNO 발생, 가해자/피해자, `dBNOId`
- `LogPlayerKillV2`: 확킬/킬 판정, `dBNOMaker`, `finisher`, `killer`, assists
- `LogPlayerRevive`: 부활자/피부활자, `dBNOId`
- `LogGameStatePeriodic`: 자기장/안전구역 상태
- `LogPhaseChange`: 페이즈 전환
- `LogItemUse`: 회복/부스트 등 아이템 사용

### Optional Events
코드 또는 특정 telemetry 원본에서 발견되더라도 공식 문서에 명시되지 않은 이벤트는 optional로 취급합니다. 예를 들어 projectile hit/explosion 계열 이벤트가 존재하더라도, core 지표는 `LogPlayerUseThrowable`과 `LogPlayerTakeDamage`로 계산하고 optional 이벤트는 보조 증거로만 사용합니다.

## 3. Official Field Notes

모든 텔레메트리 이벤트는 공통적으로 다음 필드를 가집니다.
- `_D`: 이벤트 발생 시각
- `_T`: 이벤트 타입
- `common`: 게임 진행 상태 정보

### `LogPlayerKillV2`
중요 필드:
- `dBNOId`: DBNO와 kill 연결에 사용할 수 있는 식별자
- `victim`: 피해자
- `dBNOMaker`: 기절시킨 사람
- `finisher`: 확킬한 사람
- `killer`: 최종 킬 판정자
- `assists_AccountId`: 어시스트 accountId 목록
- `isSuicide`: 자살 여부

사용 원칙:
- 기절 기여와 확킬 기여를 분리합니다.
- 트레이드 계산은 가능하면 `dBNOId`, `dBNOMaker`, `finisher`를 함께 봅니다.

### `LogPlayerMakeGroggy`
중요 필드:
- `attacker`: 기절시킨 사람
- `victim`: 기절한 사람
- `damageTypeCategory`, `damageReason`, `damageCauserName`
- `distance`
- `dBNOId`

사용 원칙:
- 아군이 누가/누구에게 기절했는지 저장합니다.
- 이후 10초 내 제압 이벤트와 연결할 때, "아군을 눕힌 바로 그 적"인지 확인해야 합니다.

### `LogPlayerTakeDamage`
중요 필드:
- `attackId`
- `attacker`, `victim`
- `damageTypeCategory`, `damageReason`, `damageCauserName`
- `damage`
- `isThroughPenetrableWall`

사용 원칙:
- 선제 타격, 반격 지연, 피해 기반 유틸리티 결과의 core source입니다.
- 수류탄/화염병/섬광 등 유틸리티 결과는 공식적으로 이 이벤트의 피해 원인 필드로 확인합니다.
- **[V11.9.2]**: 자기장 피해량은 HP 단위로 누적 집계하며, 경계선 50m(Edge Play)/200m(Fatal Delay) 판정의 근거로 사용함.

### `LogPlayerUseThrowable`
중요 필드:
- `attackId`
- `attacker`
- `attackType`
- `weapon`

사용 원칙:
- 투척물 사용 시점과 사용자를 잡는 core source입니다.
- 실제 효과는 `LogPlayerTakeDamage`와 `attackId`, 시간창, 거리창으로 연결합니다.
- 연막 세이브는 공식 원본 필드가 아니라 BGMS 파생 지표입니다.

### `LogPlayerPosition`
중요 필드:
- `character`
- `vehicle`
- `elapsedTime`
- `numAlivePlayers`

사용 원칙:
- 분석 엔진에서는 원본 cm 좌표를 m 단위로 환산해 3D 거리 계산에 사용합니다.
- 지도 리플레이에서는 0-8192 좌표로 정규화해 Leaflet/Canvas 렌더링에 사용합니다.
- 비행/낙하 노이즈는 `z < 10m` 또는 매치 시작 후 120초 이후 기준으로 필터링합니다.
- **[V11.9.2]**: 모든 위치 데이터는 수집 즉시 미터(m) 단위로 정규화 완료됨.

### `LogGameStatePeriodic`
사용 원칙:
- `safetyZonePosition` / `safetyZoneRadius`: White Zone, 다음 안전구역
- `poisonGasWarningPosition` / `poisonGasWarningRadius`: Blue Zone, 현재 자기장 경계
- zone moving 판정은 `blueRadius > whiteRadius * 1.05`를 기준으로 합니다.
- **[V11.9.2]**: 모든 구역 좌표 및 반경은 미터(m) 단위로 정규화됨.

## 4. Derived Tactical Metrics

아래 지표는 PUBG 공식 원본 필드가 아니라 BGMS가 공식 이벤트를 기반으로 계산하는 derived metrics입니다.

### Initiative Rate
- 의미: 유저가 먼저 유효 피해를 준 교전 세션 중 제압으로 이어진 비율
- Core events: `LogPlayerTakeDamage`, `LogPlayerMakeGroggy`, `LogPlayerKillV2`
- 주의: "먼저 쐈다"가 아니라 "먼저 유효 피해를 줬다"에 가깝습니다.

### Trade Latency / Trade Success Rate
- 의미: 아군이 기절한 뒤, 해당 아군을 눕힌 적을 내가 제압하기까지 걸린 시간/성공률
- Core events: `LogPlayerMakeGroggy`, `LogPlayerKillV2`
- 권장 연결: `dBNOId` 우선, 없으면 attacker/victim name/accountId와 시간창 보조
- 측정 실패 시 0이 아니라 "측정 불가"로 전달합니다.
- **[V11.9.2]**: 아군 기절 후 30초 내 복수 성공 여부를 비율(%)로 환산함.

### Counter-Attack Latency / Cover Rate
- 의미: 내가 피해를 받은 뒤 같은 교전 상대에게 반격 피해를 주기까지의 시간/성공률
- Core events: `LogPlayerTakeDamage`
- `coverRateSampleCount`가 0이면 측정 불가입니다.

### Isolation Index
- 의미: 교전/기절 시점에서 아군과의 거리, 적과의 거리, 고도차, 십자포화 노출을 반영한 BGMS 공간 지표
- Core events: `LogPlayerPosition`, `LogPlayerMakeGroggy`, `LogPlayerKillV2`
- 분석 단위는 **m**입니다.
- DBNO 시점 샘플을 우선하고, 없을 때 이동 중 position 샘플을 보조로 사용합니다.

### Utility Value
- 의미: 투척물/유틸리티 사용이 피해, 시야 차단, 구조, 진입 차단에 기여했는지 평가
- Core events: `LogPlayerUseThrowable`, `LogPlayerTakeDamage`, `LogPlayerMakeGroggy`, `LogPlayerRevive`
- 연막 세이브는 아군 DBNO 시점/위치와 연막 사용 시점/위치를 연결한 파생 지표입니다.
- 투척물 적중은 공식 문서 기준 `LogPlayerTakeDamage`에 근거한 "투척물 피해 발생"으로 표현하는 것이 가장 안전합니다.
- **[V11.9.2]**: 섬광탄 적중(Stun Hits)은 `combatPressure.stunHits`를 통해 실제 적중 횟수를 별도 집계함.

### Golden Time Damage
- 의미: 생존 구간별 피해량
- Core event: `LogPlayerTakeDamage`
- 구간: 0-5분, 5-15분, 15-25분, 25분+

### Zone Adaptation
- 의미: 자기장 피해량, edge play, fatal delay 등 운영 지표
- Core events: `LogGameStatePeriodic`, `LogPlayerTakeDamage`, `LogPhaseChange`
- **[V11.9.2]**: 50m(Edge Play), 200m(Fatal Delay) 임계값을 적용하여 정밀하게 횟수를 집계함.

## 5. Caching and Versioning Rules

- PUBG 공식 Usage Guide는 반복 API 요청을 줄이기 위해 캐싱 레이어를 권장합니다.
- BGMS는 분석용 캐시와 지도용 캐시를 분리합니다.
- 분석 캐시: `${matchId}_analyze.json`
- 지도 캐시: `${matchId}_${accountId}_${mode}_map.json`
- 지도 캐시는 팀 식별과 mode에 따라 응답이 달라지므로 matchId 단독으로 캐시하면 안 됩니다.
- 전술 계산 산식이 바뀌면 `RESULT_VERSION`을 올려 `processed_match_telemetry`가 재생성되도록 해야 합니다.
- **[V11.9.2]**: 최신 정규화 산식이 적용된 버전입니다.

## 6. AI Prompting Rules

- AI에는 공식 원본 이벤트명과 BGMS 파생 지표를 구분해서 전달합니다.
- `0`과 "측정 불가"를 혼동하지 않습니다.
- 표본 수가 있는 지표는 `sampleCount`를 함께 전달합니다.
- Apple-to-Apple 비교는 gameMode, FPP/TPP, squad/solo, ranked/normal 조건을 최대한 맞춘 벤치마크로 수행합니다.
- 공식 문서에 없는 이벤트를 근거로 단정하지 않습니다.
- 연막 세이브, 고립도, 압박 지수 등은 "BGMS 파생 분석"임을 전제합니다.
- **[V11.9.2]**: 모든 물리 지표를 정규화된 단위로 전달하며, 유저 친화적 게임 용어(양각 노출 등) 사용을 권장함.

## 8. Backend & Frontend Collaboration Guide (V11.9.2)

### [Backend] `lib/pubg-analysis/processor.ts`
- **역할**: 텔레메트리 파싱 및 단위 정규화의 Source of Truth.
- **주의**: 새로운 거리 기반 지표 추가 시 반드시 `/ 100` 처리를 완료하여 내보낼 것.

### [API/엔진] `app/api/pubg/ai-summary/route.ts`
- **역할**: 다수 경기의 지표 합산 및 AI 프롬프트 생성.
- **주의**: `minDist`, `heightDiff` 등을 합산할 때 이미 미터 단위이므로 다시 100으로 나누지 말 것.

### [Frontend] `components/stat/`
- **역할**: 시각화 및 리포트 렌더링.
- **주의**: 백엔드에서 정규화된 값이 오므로, `minDist` 옆에는 `m`, `bluezoneWaste` 옆에는 `HP` 단위를 붙여서 렌더링함.
