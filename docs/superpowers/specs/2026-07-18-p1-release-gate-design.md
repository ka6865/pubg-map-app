# P1 통합 배포 게이트 설계

- 작성일: 2026-07-18
- 기준 브랜치: 로컬 `develop`
- 기준 보고서: `docs/reviews/2026-07-15-feature-code-review.md`
- 운영 제약: Vercel, Supabase, Cloudflare R2 무료 플랜

## 1. 결정

현재 완료된 P0·Hotdrop·텔레메트리 작업은 로컬 `develop`에 먼저 통합하되 즉시 `main`에 반영하지 않는다. 보고서의 P1 8건을 모두 구현하고 통합 검증을 통과한 뒤 `develop → main` 단일 배포 후보로 올린다. P2 작업은 해당 배포가 안정화된 이후 별도 작업 주기로 시작한다.

## 2. 선택 근거

검토한 선택지는 다음 두 가지다.

1. P0·Hotdrop·텔레메트리를 먼저 배포하고 P1을 후속 배포한다.
   - 장점: P0 수정이 운영에 빠르게 반영된다.
   - 단점: 현재 누적 변경이 이미 크고, 짧은 기간에 두 번의 migration·배포·회귀 검증이 필요하다.
2. 현재 작업과 P1 전체를 하나의 배포 후보로 묶고 P2를 분리한다.
   - 장점: 보안 경계를 한 번에 정합화하고 P2 구조 개선이 배포 후보에 섞이는 것을 막는다.
   - 단점: P1 완료 전까지 `main` 반영이 지연되므로 로컬 브랜치 관리와 회귀 범위 통제가 필수다.

사용자 결정에 따라 2번을 채택한다. 위험을 줄이기 위해 P1은 한 덩어리로 구현하지 않고 기능별 테스트·한글 커밋·문서 갱신 단위로 나눈다.

## 3. 브랜치와 커밋 경계

1. `codex/telemetry-contract-identity`의 코드·문서·검증을 완료한다.
2. 텔레메트리 브랜치를 로컬 `develop`에 통합한다.
3. 각 P1 항목은 독립 작업 브랜치 또는 격리된 worktree에서 구현한다.
4. 각 항목은 실패 테스트, 최소 구현, 관련 회귀 테스트, 정적 검증, 한글 커밋, 보고서 갱신을 하나의 완료 단위로 삼는다.
5. P1 8건이 모두 완료되기 전에는 `main` PR을 생성하지 않는다.
6. P1 배포 후보에는 P2 기능 변경을 포함하지 않는다.

사용자 백업 문서와 관련 없는 기존 변경은 stage, 수정, 삭제하지 않는다.

## 4. P1 구현 순서

보안 영향과 무료 플랜 자원 보호를 기준으로 다음 순서를 사용한다.

1. 비회원 글·댓글 Turnstile 서버 결합과 요청 제한
2. 게시글 Storage 삭제 소유권·참조 검증
3. 제보 알림 임계값 전이·중복 발송 원자화
4. 관리자 승인 작업 원자적 claim·중복 실행 차단
5. analytics 서버 권위 필드·batch 상한·요청 제한·보존 정책
6. PUBG refresh TTL·single-flight·요청 제한
7. AI squad 장애 응답에서 측정되지 않은 fallback 제거
8. 미지원 맵 3D 리플레이 차단과 capability registry 통합

## 5. 무료 플랜 운영 원칙

- Vercel 요청 시간과 실행 횟수를 늘리는 별도 상시 작업을 추가하지 않는다.
- GitHub Actions로 이전한 Hotdrop 작업을 Vercel Cron으로 되돌리지 않는다.
- 요청 제한은 우선 기존 Supabase와 애플리케이션 캐시를 이용하고 외부 유료 서비스를 추가하지 않는다.
- PUBG API 호출은 nickname·platform·기능 단위 TTL과 single-flight로 중복을 억제한다.
- analytics와 guest write에는 batch·본문 크기·빈도 상한을 둬 Supabase 무료 용량을 보호한다.
- R2 자동 삭제는 안전한 조건부 삭제 계약이 입증될 때까지 비활성화하고 저장량을 모니터링한다.
- 실제 Chrome 회귀 중 동일 실패 요청을 반복해 PUBG API 예산이나 Discord 알림을 소모하지 않는다.

## 6. 통합 검증 게이트

P1 전체 완료 후 다음 증거가 모두 있어야 `main` PR을 생성한다.

1. 각 P1 항목의 공격·경쟁·실패 경계를 재현하는 회귀 테스트
2. `npm run verify:core`
3. `npm run verify:analysis`
4. `npm run verify:admin`
5. 전체 Vitest와 Jest
6. 신규·변경 migration의 임시 PostgreSQL 적용 검증
7. 환경변수 값 노출 없는 설정 누락·오설정 fail-closed 검증
8. 실제 Chrome에서 지도, 게시판 guest write, 관리자 승인, 전적 검색, 2D·3D 리플레이 회귀
9. Steam 전적 검색은 `KangHeeSung_`을 사용하되 동일한 외부 실패 요청을 반복하지 않음
10. 변경 파일의 호출부·선언부 타입/인자 전수 검사, import 경로 검사, 신규 `console.log`·unused import·주석 처리 코드 부재 확인
11. 보고서의 P1 상태·검증 수치·남은 P2 항목 최신화
12. `git diff --check`와 작업 트리 범위 확인

운영 Supabase migration은 배포 순서와 롤백 경계를 별도로 확인한 뒤 적용한다. 로컬 테스트 통과만으로 운영 migration 적용 완료를 주장하지 않는다.

## 7. 배포 순서

1. P1 완료 커밋이 모두 포함된 `develop` 배포 후보를 고정한다.
2. migration 선적용이 필요한 변경과 애플리케이션 선적용이 가능한 변경을 구분한다.
3. 필요한 Vercel 환경변수와 GitHub Actions Secrets의 이름·연결 관계만 검증하고 값은 출력하지 않는다.
4. `develop → main` PR에서 전체 변경과 배포 순서를 재검토한다.
5. `main` 병합 후 Vercel 배포와 GitHub Actions 상태를 확인한다.
6. 운영 회귀가 확인된 뒤 P2 작업을 시작한다.

## 8. 실패와 중단 기준

- 전체 게이트 중 하나라도 실패하면 `main` PR을 생성하지 않는다.
- 실제 Chrome의 HTTP 500은 정제된 오류만으로 원인을 추측하지 않고 서버 단계별 안전 로그와 재현 테스트로 범위를 좁힌다.
- migration과 코드의 선후 관계가 확정되지 않으면 배포를 중단하고 로컬 검증을 계속한다.
- 무료 플랜 한도에 영향을 줄 수 있는 반복 호출·전체 R2 삭제·운영 DB 전체 스캔은 수행하지 않는다.
- P2 개선은 P1 결함을 우회하기 위한 임시 해결책으로 끌어오지 않는다.

## 9. 완료 기준

P1 8건의 코드·테스트·한글 커밋·보고서 갱신이 모두 존재하고, 통합 검증 게이트가 최신 실행에서 통과하며, 배포 순서와 환경변수 준비 상태가 확인됐을 때만 P1 배포 후보가 완료된 것으로 판단한다. `main` 병합과 운영 회귀 확인 전에는 P1 배포 완료로 표시하지 않는다.
